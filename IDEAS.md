# IDEAS.md

This document serves as an evolving brainstorm and think-tank for the CLIProxyAPI Plus project, detailing extensive architectural overhauls, missing functionalities, potential pivots, and granular codebase improvements.

## Architectural Improvements & Refactoring
1. **Dynamic Plugin Architecture (`.so` / WASM)**: Currently, adding a new provider (e.g., Kiro, Anthropic, Qwen) requires recompiling the binary. We could refactor the `internal/translator/` module to load standard Go Plugins (`.so` files) or WebAssembly (WASI) modules at runtime. This creates a marketplace where the community can develop custom translators that drop directly into the `config.yaml`.
2. **Move to a Database-Backed State**: Currently, state heavily relies on `config.yaml` and hot-reloading configurations in memory. Moving purely dynamic data (API keys, quotas, model alias states) into an embedded database like SQLite (via `modernc.org/sqlite` or `gorm`) would drastically improve concurrent read/write scaling, reduce file locking complexity, and prepare the project for HA (High Availability) clustering.
3. **gRPC Internal Service Mesh**: If the proxy is deployed in a cluster, components could communicate over gRPC instead of standard REST. The UI could speak directly to the backend via a generated gRPC-Web client, which provides type-safe boundaries and reduces HTTP overhead.
4. **Port the Engine to Rust**: For maximum performance and near-zero memory footprint (if that becomes the strict project goal), porting the core HTTP routing engine and JSON schema translators to Rust (using `hyper`/`tokio` and `serde_json`) could lower latency for high-throughput enterprise deployments. Go remains excellent for the current scope, however.

## New Feature Horizons
1. **Cost & Token Budgeting per Sub-Tenant**: Introduce logical "Tenants" or "Teams" into the proxy config. Allow admins to create child API keys generated by the proxy itself, capping them to specific dollar amounts (calculated in real-time based on `internal/registry` model pricing metrics) or token limits per month.
2. **Streaming Prompt Caching Optimization**: Anthropic and Google Gemini support Prompt Caching. The proxy should detect identical system prompt / context prefixes across different requests and automatically inject caching headers or flags to drastically reduce upstream token costs.
3. **Advanced Load Balancing (Latency-Based Routing)**: Extend `routingStrategy` beyond `round-robin` and `priority` to include `latency-based` and `cost-based`. The proxy could ping provider endpoints asynchronously to track response latency, automatically routing traffic to the fastest responding key or the cheapest compatible model.
4. **Multi-Modal Native Support**: Build out robust Voice (Audio-to-Audio) API endpoints as they become standard. Integrate support for streaming video to models that accept continuous frame inputs.

## UI/UX Innovations
1. **Interactive Node-based Routing Graph**: Inside the React UI, instead of just list configurations, introduce a `ReactFlow` node graph where admins can visually draw lines connecting Client -> Proxy -> Provider -> API Key, defining priorities and fallbacks visually.
2. **Live Request Debugger (Wireshark for LLMs)**: A real-time console in the UI that streams incoming requests and outgoing payloads side-by-side, highlighting exactly what the proxy translator modified (e.g. visualizing function calling structures being morphed into XML tags).
3. **Analytics Dashboard Revamp**: Integrate interactive charts showing usage spikes, error rates by provider, and latency percentiles over time.

## Project Pivot Concepts
1. **Enterprise API Gateway for AI**: Shift marketing and feature development away from a personal "CLI" proxy to an Enterprise API Gateway (similar to Kong or Apigee but strictly for LLMs), introducing features like SAML/SSO admin login, Data Loss Prevention (DLP) to scrub PII from prompts before they leave the network, and full OpenTelemetry tracing.
2. **Local Fallback Mode (Local-First AI)**: Deeply integrate with Ollama, LM Studio, or llama.cpp. If a remote API key fails or errors out, the proxy automatically falls back to a locally hosted model on the user's machine to answer the prompt, ensuring 100% uptime.
